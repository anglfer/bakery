Módulo: Login / Autenticación  
Reglas de negocio  
• Los clientes pueden registrarse de manera autónoma  a través d e la opción “Crear 
cuenta”  disponible en el módulo de login.  Campos: id_cliente, nombre, apellido, 
teléfono, usuario_cliente, password_hash, fecha_registro, estado_cliente .  
• Los empleados no pueden registrarse por sí mismos; su registro debe ser realizado 
exclusivamente por un usuario con rol de administrador.  
• Todo usuario debe autenticarse mediante credenciales válidas (usuario y contraseña) 
para acceder al sistema.  
• Las contraseñas deben cumplir con una política de seguridad que incluya:  
✓ Mínimo 8 caracteres  
✓ Al menos una letra mayúscula  
✓ Al menos una letra minúscula  
✓ Al menos un número  
✓ Al menos un carácter especial  
• El sistema debe validar las contraseñas contra listas de contraseñas inseguras para 
evitar combinaciones vulnerables.  
• El sistema debe implementar un mecanismo de CAPTCHA en el inicio de sesión para 
prevenir accesos automatizados.  
• El sistema debe implementar autenticación de dos factores (2FA), generando un 
código de un solo uso con tiempo de expiración limitado  (5 minutos). 
• Se permite un máximo de 3 intentos fallidos de inicio de sesión; al exceder este límite, 
la cuenta debe ser bloqueada temporalmente.  
• El sistema debe registrar todos los intentos de inicio de sesión, exitosos y fallidos  
deberán registrarse en un archivo de logs o en base de dato  (no sé dónde es que iban 
a usar mongo). Los datos mínimos deberán ser el error presentado y la fecha .  
• Los usuarios deben cambiar su contraseña de manera periódica según la política 
definida por el sistema.  
• Las contraseñas deben almacenarse de forma encriptada y nunca en texto plano.  
• El sistema debe mostrar al usuario la información de su último inicio de sesión.  
• Las sesiones de usuario deben tener una duración máxima de 10 minutos; al expirar, 
el usuario deberá autenticarse nuevamente.  
• No se debe permitir el acceso a usuarios inactivos o bloqueados.  
• El acceso a las funcionalidades del sistema debe estar controlado según el rol del 
usuario (principio de mínimo privilegio).  
• El sistema no debe mostrar mensajes de error que revelen información sensible (por 
ejemplo, si el usuario o la contraseña son incorrectos de forma específica).

Módulo: Gestión de roles  
Reglas del negocio  
• Solo los usuarios con rol de Administrador  tienen la autoridad para crear, modificar 
o desactivar roles dentro del sistema.  
• Cada rol debe contar con un nombre único que lo identifique dentro del sistema.  
• Todo rol debe definir de manera explícita los permisos y acciones que puede realizar 
en cada módulo del sistema.  
• Los roles base del sistema ( Administrador, Ventas y Producción ) son obligatorios y 
no pueden ser eliminados ni desactivados.  
• Todo usuario debe tener un rol asignado para poder acceder y operar dentro del 
sistema.  
• Los permisos deben asignarse bajo el principio de mínimo privilegio , otorgando 
únicamente los accesos necesarios según la función del usuario.  
• Los cambios realizados en los permisos de un rol deben aplicarse de forma 
inmediata a todos los usuarios que tengan dicho rol asignado.  
• No se permite la eliminación física de roles; únicamente se permite su desactivación 
lógica para conservar la trazabilidad. Y también puden activarse cuando sea 
necesario.  
• Un rol en estado inactivo no puede ser asignado a nuevos usuarios.  
• La definición de roles debe estar alineada con las funciones reales del personal 
dentro de la organización.  
• El sistema debe registrar la fecha de creación y el estado del rol (activo/inactivo).  
• El sistema debe permitir la asignación de permisos por módulo y tipo de acción 
(lectura, escritura, actualización, eliminación).  
• No se permite acceder al módulo sin autenticación.  
• Validar que el nombre del rol no exista previamente.  
• Validar que el rol tenga al menos un permiso asignado.  
• Validar coherencia en la asignación de permisos.  
• Verificar que no se modifiquen roles base críticos.

materia prima.

dentro del sistema.  
• La información de los proveedores debe mantenerse actualizada para garantizar el 
correcto abastecimiento de insumos.  
• El sistema debe permitir la consulta del historial de compras asociadas a cada 
proveedor.  
• La relación entre proveedores y compras  de inventario  debe garantizar la trazabilidad 
de todas las operaciones realizadas.  
• El sistema debe registrar la fecha de alta del proveedor y su estado (activo/inactivo).  
• El correo electrónico del proveedor debe cumplir con un formato válido.  
• Validar que el proveedor no exista previamente.

• Validar campos obligatorios (nombre, contacto, etc.).  
• Validar formato de teléfono (si aplica).  
• Confirmar registro o actualización exitosa.

Módulo: Inventario de Materia Prima  
Reglas del negocio  
• Toda materia prima debe registrarse obligatoriamente con una unidad base para su 
control interno.  
• La unidad de compra puede ser distinta a la unidad base; sin embargo, debe existir un 
factor de conversión definido.   

• El factor de conversión debe ser mayor a cero y válido para garantizar cálculos 
correctos.  
• El sistema debe almacenar todas las cantidades en unidad base para mantener la 
consistencia del inventario.  
• El inventario de materia prima no puede presentar valores negativos bajo ninguna 
circunstancia.  
• Toda entrada de materia prima debe originarse exclusivamente a partir de una compra 
registrada en el sistema.  
• Toda salida de materia prima debe estar asociada a un proceso de producción o a un 
ajuste autorizado.  
• El sistema debe registrar todos los movimientos de inventario (entradas, salidas y 
ajustes) para garantizar la trazabilidad.  
• El porcentaje de merma debe aplicarse automáticamente en los cálculos de consumo 
de materia prima durante la producción.  
• El sistema debe generar alertas cuando la cantidad disponible de una materia prima 
sea menor al stock mínimo definido.  
• No se permite la eliminación física de materias primas; únicamente se permite su 
desactivación lógica. Y se puede volver a activar.  
• Las materias primas en estado inactivo no podrán ser utilizadas en nuevos procesos de 
producción ni en compras.  
• El sistema debe garantizar la disponibilidad de insumos necesarios para la operación 
diaria mediante el control del inventario.  
• Cada materia prima debe ser identificable de manera única dentro del sistema.   
• Las materias primas inactivas no deben utilizarse en nuevos procesos de producción.  
• El control del inventario debe permitir asegurar la disponibilidad de insumos para la 
operación diaria.  
• Se debe generar una alerta cuando el stock sea menor al mínimo.  
• No se permite acceder al módulo sin autenticación.  
• Todas las operaciones sobre inventario deben validarse en el backend para evitar 
manipulación indebida .  
• Validar que la materia prima exista.  
• Validar unidad base definida.  
• Validar factor de conversión.  
• Validar cantidades numéricas válidas.  
• Confirmar actualización correcta del inventario.  
• Generar alertas de stock mínimo.  
• Mostrar mensajes claros al usuario.  

Módulo: Compra de Materia Prima  
Reglas del negocio  
• Toda compra debe registrarse mediante una estructura de cabecera y detalle.  
• Cada compra debe contar obligatoriamente con al menos un registro en el detalle.  
• El identificador de la compra (id_compra) debe generarse automáticamente por el 
sistema.  
• Al comprar materia prima se deberá de modificar la existencia adecuando la cantidad 
en la medida correcta (Ej. 1 bulto de harina → 50 kilogramos → 50,000 gramos). En cada 
materia prima deberá de contener un % de merma ya sea por producción, por manejo 
o por desperdicio.  
• La fecha de compra es obligatoria y debe registrarse al momento de crear la compra.  
• El usuario que registra la compra debe quedar asociado a la operación para garantizar 
trazabilidad.  
• Solo se pueden registrar compras con proveedores en estado activo.  
• No se permite seleccionar proveedores inexistentes o desactivados.  
• El proveedor debe existir previamente en el sistema.  
• Cada registro del detalle debe incluir: Materia prima válida , Unidad de compra válida , 
Cantidad mayor a cero , Precio unitario mayor o igual a cero  
• El subtotal de cada detalle debe calcularse automáticamente:  subtotal = 
cantidad_comprada × precio_unitario  
• El total de la compra debe calcularse automáticamente como la suma de subtotales.  
• No se permite capturar manualmente el total de la compra.  
• La conversión de unidades a unidad base es obligatoria en cada registro de detalle.  
• La cantidad base debe calcularse automáticamente utilizando el factor de conversión 
definido en la materia prima.  
• El factor de conversión debe ser válido y mayor a cero.  
• Al guardar la compra, el sistema debe actualizar automáticamente el inventario:  
Incrementando la cantidad disponible , Registrando un movimiento tipo ENTRADA  . 
• No se permite la eliminación ni modificación de compras registradas (solo consulta).  
• Toda compra debe registrarse inicialmente con estado de pago PENDIENTE .  
• El estado de pago puede actualizarse a PAGADO  posteriormente.  
• El registro del pago debe vincularse con el módulo de salida de efectivo  Modulo de 
ventas.  
• Todos los movimientos generados deben garantizar la trazabilidad de la operación.  
• Validar existencia y estado del proveedor.  
• Validar existencia de materia prima.  

• Validar cantidad > 0.  
• Validar precio ≥ 0.  
• Validar unidad de compra válida.  
• Validar cálculo correcto de subtotales.  
• Validar cálculo correcto del total.  
• Validar conversión a unidad base.  
• Validar factor de conversión.  
• Validar actualización de inventario.  
• Confirmar registro exitoso de compra.  
• Confirmar actualización del inventario.  
• Mostrar mensajes claros al usuario.  
 
Modulo Explosión de Materiales (Recetas)  
Reglas del negocio  
• Toda receta debe estar obligatoriamente asociada a un producto existente en el 
sistema.  
• Cada receta debe registrarse mediante una estructura de cabecera y detalle.  
• El identificador de la receta (id_receta) debe generarse automáticamente.  
• El sistema contará inicialmente con un catálogo de 10 estilos de pasteles predefinidos 
(chocolate, red velvet, vainilla, zanahoria, chocoflan, cheesecake, moka, frutos secos, 
tres leches y pastel helado de Oreo). El alta de nuevos estilos de productos esta rá 
restringida al administrador.  
• Cada receta debe definir un rendimiento base , el cual indica la cantidad de producto 
que se obtiene al ejecutar la receta.  
• Todas las cantidades de materia prima en el detalle de la receta deben definirse en 
unidad base.  
• Una receta puede contener múltiples materias primas asociadas.  
• El sistema debe calcular automáticamente la explosión de materiales, es decir, la 
cantidad de insumos requeridos en función de la cantidad a producir.  
• Antes de iniciar un proceso de producción, el sistema debe validar la disponibilidad de 
materia prima en inventario.  
• No se permitirá iniciar producción si no existe suficiente materia prima disponible.  
• En caso de insuficiencia de insumos, el sistema debe mostrar alertas claras indicando 
los faltantes.  

• Una receta puede tener múltiples versiones; sin embargo, solo una versión puede estar 
activa por producto.  
• Al modificar una receta, se debe generar una nueva versión, conservando el historial 
de versiones anteriores.  
• No se permite la eliminación física de recetas; únicamente se permite su desactivación 
lógica. Y se puede volver a activar.  
• Toda producción debe basarse obligatoriamente en una receta activa.  
• Al iniciar un proceso de producción, el sistema debe descontar automáticamente del 
inventario las materias primas según la explosión de materiales calculada.  
• El consumo de materia prima debe considerar el porcentaje de merma definido en el 
sistema.  
• Al finalizar la producción, los productos elaborados deben agregarse automáticamente 
al inventario de productos terminados.  
• El sistema debe permitir la actualización controlada de recetas únicamente por 
usuarios autorizados.  
• El área de ventas debe poder visualizar los productos en proceso de producción y las 
cantidades disponibles.  
• El área de ventas podrá generar solicitudes de producción en función de la demanda 
detectada.  
• El acceso al módulo debe estar restringido por roles (Administrador y Producción) . 
• Validar existencia del producto.  
• Validar existencia de materias primas.  
• Validar cantidades mayores a cero.  
• Validar unidad base.  
• Validar rendimiento base > 0.  
• Confirmar registro o actualización de receta.  
• Confirmar cálculo de insumos.  
• Confirmar descuento de inventario.  
• Confirmar actualización de productos terminados.  
• Mostrar alertas en caso de faltantes.  
• Validar receta activa antes de producción.  
• Validar disponibilidad de inventario.  
• Validar cálculo correcto de explosión de materiales.  
• Validar aplicación de merma.  
• Validar consistencia en versiones.  
 

Módulo de Producción  
Reglas del negocio  
• Toda orden de producción debe registrarse con una estructura definida que incluya:  
id_orden (automático) , receta asociada , producto , cantidad a producir , fechas y 
usuario responsable .  
• El identificador de la orden (id_orden) debe generarse automáticamente.  
• Toda orden de producción debe estar asociada obligatoriamente a una receta activa.  
• Antes de iniciar la producción, el sistema debe validar: Que la receta esté activa , Que 
exista suficiente materia prima en inventario , Toda orden de producción debe iniciar en 
estado PENDIENTE.  
• Las órdenes de producción deben manejar los siguientes estados: PENDIENTE , 
EN_PROCESO , FINALIZADO y CANCELADO  
• Al cambiar el estado a EN_PROCESO, el sistema debe: Calcular automáticamente el 
consumo de materia prima , Aplicar el porcentaje de merma , Descontar los insumos 
del inventario .  
• El consumo de materia prima debe calcularse con la fórmula: cantidad_real = 
cantidad_receta × cantidad_producir × (1 + porcentaje_merma / 100)  
• Todo descuento de materia prima debe registrarse como un movimiento tipo SALIDA 
en el inventario.  
• No se permitirá iniciar producción si no existe suficiente materia prima disponible.  
• Solo se puede cancelar una orden de producción si se encuentra en estado 
PENDIENTE.  
• No se permite cancelar órdenes en estado EN_PROCESO o FINALIZADO.  
• Al finalizar una orden de producción (estado FINALIZADO), el sistema debe:  
Incrementar automáticamente el inventario de productos terminados , Registrar el 
movimiento correspondiente .  
• Toda orden de producción debe quedar registrada para consulta y trazabilidad.  
• El sistema debe permitir visualizar el detalle de la producción .   
• El sistema debe calcular automáticamente el costo de producción con base en los 
insumos utilizados.  
• El área de ventas podrá consultar las órdenes de producción y la disponibilidad de 
productos, pero no podrá modificarlas.  
• El sistema debe garantizar la trazabilidad completa desde el consumo de insumos 
hasta la generación del producto terminado.  
• Validar existencia de receta.  
• Validar receta activa.  

• Validar cantidad a producir > 0.  
• Validar usuario autorizado.  
• Validar disponibilidad de materia prima.  
• Validar cálculo correcto de consumo.  
• Validar aplicación de merma.  
• Validar transición correcta de estados.  
• Confirmar creación de orden.  
• Confirmar descuento de inventario.  
• Confirmar incremento de producto terminado.  
• Mostrar mensajes claros en caso de error.

• Toda solicitud debe iniciar en estado PENDIENTE .  
• Las solicitudes deben manejar los siguientes estados: PENDIENTE , APROBADA y 
RECHAZADA  
• Solo el área de producción  puede aprobar o rechazar solicitudes.  
• Una solicitud en estado APROBADA  puede utilizarse como base para generar una 
orden de producción.  

• Las solicitudes en estado RECHAZADA  no deben generar órdenes de producción.  
• El sistema debe permitir la vinculación entre la solicitud y la orden de producción 
mediante el identificador correspondiente.

PENDIENTE .  
• Una vez que la solicitud ha sido aprobada o rechazada, no puede ser modificada.  
• Todas las solicitudes deben almacenarse para consulta y trazabilidad.  
• El sistema debe permitir consultar solicitudes por estado, fecha y producto.  
• El sistema debe registrar el usuario responsable tanto de la solicitud como de su 
aprobación o rechazo.  
• Las cantidades solicitadas deben ser mayores a cero.  
• El sistema debe permitir registrar observaciones tanto en la solicitud como en su 
resolución.

Modulo Costo y Utilidad de Producto  
Reglas del neg ocio  
• Todo producto debe tener obligatoriamente una receta asociada  para poder calcular 
su costo.  
• El costo de producción debe calcularse con base en las materias primas definidas 
en la receta.  
• El sistema debe calcular el costo de cada ingrediente considerando: Cantidad 
requerida , Costo unitario de la materia prima y Porcentaje de merma  

• La cantidad real de materia prima debe calcularse automáticamente aplicando 
merma:  cantidad_real = cantidad_requerida × (1 + porcentaje_merma / 100)  
• El costo por ingrediente debe calcularse automáticamente: costo_ingrediente = 
cantidad_real × costo_unitario  
• El costo total de producción debe ser la suma de todos los costos de ingredientes.  
• El sistema debe calcular la utilidad del producto: utilidad = precio_venta − 
costo_produccion .  
• El porcentaje de utilidad debe calcularse automáticamente: porcentaje_utilidad = 
(utilidad / precio_venta) × 100  
• El cálculo de costos debe actualizarse automáticamente cuando: Cambie el costo 
de una materia prima , Se modifique una receta , Se actualice el precio de venta .  
• No se permite modificar manualmente el costo de producción.  
• El sistema debe permitir consultar el desglose de costos por ingrediente.  
• El sistema debe mostrar la utilidad y porcentaje de ganancia por producto.  
• Si un producto no tiene receta activa, no se podrá calcular su costo.  
• Si alguna materia prima no tiene costo definido, no se podrá realizar el cálculo.  
• El sistema debe trabajar con valores monetarios en pesos mexicanos (MXN) . 
• No se permite acceder al módulo sin autenticación.  
• No se permite registrar precios de venta negativos o en cero.
