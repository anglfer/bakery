Página 2 de 63 Introducción  
El presente documento corresponde a la documentación técnica y funcional del sistema SoftBakery, 
una plataforma web de gestión integral diseñada para automatizar y optimizar las operaciones de 
una pastelería. El sistema abarca módulos de inventario, produc ción, ventas, compras, pedidos en 
línea y administración de usuarios.  
Este documento está organizado en tres grandes secciones: Descripciones de Módulos, que detalla 
los requisitos funcionales de cada componente del sistema; Diseños de Módulos, que muestra las 
interfaces gráficas de usuario diseñadas; e Historias de Usuario,  que describen las funcionalidades 
desde la perspectiva de cada tipo de usuario. Adicionalmente se incluye el Calendario de Actividades 
del proyecto.

RF01 — Login / Autenticación  
Campo  Descripción  
ID RF01  
Nombre  Login / Autenticación  
Propósito  Garantizar que únicamente los usuarios autorizados (empleados y clientes) puedan acceder al 
sistema SoftBakery mediante un mecanismo de autenticación seguro, controlado y trazable, 
protegiendo la información de ventas, inventario y producción.  
Público Objetivo  Empleados del negocio (Administrador, Ventas, Producción) y Clientes.  
Descripción 
General  El módulo de autenticación permite validar la identidad de los usuarios mediante nombre de 
usuario y contraseña. Como medida  de seguridad, se implementa autenticación en dos factores 
(2FA) mediante un código temporal enviado al correo electrónico del usuario. El sistema asigna 
accesos y permisos según el rol del usuario.  
Características del 
Requerimiento  1.- Inicio de sesión:  Formulario con campos de usuario y contraseña, con validación de datos 
obligatorios.  
2.- Validación de credenciales:  Comparación segura mediante contraseñas cifradas (hash 
con bcrypt) sin almacenamiento en texto plano.  
3.- Autenticación en dos factores (2FA):  Generación de código OTP de 6 dígitos enviado al 
correo del usuario con vigencia de 5 minutos.  
4.- Control de intentos fallidos:  Bloqueo temporal tras 3 intentos fallidos e implementación 
de CAPTCHA para prevenir accesos automatizados.  
5.- Gestión de sesión:  Creación de sesión segura mediante cookies con expiración por 
inactividad o cierre de sesión.  
6.- Registro de actividad:  Almacenamiento de fecha y hora del último inicio de sesión, intentos 
fallidos y eventos de autenticación.  
7.- Asignación de roles:  Redirección automática según  rol (Administrador, Ventas, 
Producción). Mensajes al usuario:  Mensajes de error genéricos sin exponer información 
sensible.  
8.- Seguridad:  Implementación de protección CSRF, validación de entradas y prevención de 
ataques de fuerza bruta mediante limitación de peticiones.  
Flujo del Proceso  1.- El usuario ingresa usuario y contraseña.  
2.- El sistema valida las credenciales.  
3.- Si son correctas, se genera un código de verificación (2FA).  
4.- El código se envía al correo electrónico registrado.  
5.- El usuario ingresa el código recibido.  
6.- Si el código es válido y no ha expirado, se crea la sesión.  
7.- El sistema redirige al usuario al módulo correspondiente según su rol.  
Entradas  Ingresar nombre de usuario, contraseña, código de verificación (2FA).  
Salidas  Acceso autorizado al sistema, creación de sesión activa, mensajes de error controlados en caso 
de fallo.  
Interacción con 
otros módulos  Módulo de Usuarios (validación de credenciales y roles), Módulo de Seguridad (registro de logs, 
control de accesos), Módulo de Ventas, Producción e Inventario (acceso según permisos del 
usuario).  
Reglas de Negocio  1.- Todo usuario debe autenticarse para acceder al sistema.  
2.- El acceso se controla mediante roles (RBAC).  
3.- Se permite un máximo de 3 intentos fallidos antes de bloquear temporalmente la cuenta.  
4.- El código de verificación (2FA) tiene una vigencia de 5 minutos .  
5.- Se registra el historial de accesos del usuario.  
6.- Los usuarios inactivos o bloqueados no podrán iniciar sesión.

Página 6 de 63 RF02 — Dashboard  
Campo  Descripción  
ID RF02  
Nombre  Dashboard  
Propósito  Proveer una vista general del sistema con los indicadores más importantes del día, permitiendo 
a los usuarios tomar decisiones rápidas e informadas sobre ventas, inventario y producción en 
la pastelería SoftBakery.  
Público Objetivo  Administrador, Área de Ventas, Área de Producción  
Descripción 
General  El módulo Dashboard muestra información resumida y visual del comportamiento del negocio 
en tiempo real o del día actual, incluyendo métricas clave de ventas, productos más vendidos 
y actividad comercial. Facilita la toma de decisiones operativas mediante gráficos y tablas 
dinámicas.  
Características del 
Requerimiento  1.- Visualización de KPIs:  Total de ventas del día, número de ventas registradas y producto 
más vendido.  
2.- Gráfica de ventas por hora:  Permite identificar horas pico de venta.  
3.- Gráfica de productos más vendidos:  Representación en gráfico de barras.  
4.- Gráfica de presentaciones más vendidas : Identificación de presentaciones con mayor 
demanda.  
5.- Tabla de ventas recientes:  Incluye hora, producto, presentación, cantidad y total.  
6.- Actualización dinámica:  Información actualizada en cada carga del dashboard.  
7.- Consulta de información por día específico.  
8.- Acceso por rol:  Visualización de información según permisos del usuario.  
9.- Mensajes claros:  En caso de no haber datos, mostrar mensajes informativos.  
Flujo del Proceso  1.- El usuario inicia sesión.  
2.- Accede al dashboard según su rol.  
3.- El sistema consulta los datos del día en la base de datos.  
4.- Se calculan los indicadores (ventas, productos, etc.).  
5.- Se renderizan las gráficas y tablas.  
6.- 6. El usuario visualiza la información para la toma de decisiones.  
Entradas  Datos de ventas registradas (fecha, hora, producto, cantidad, precio), información de productos 
y presentaciones.  
Salidas  KPIs de ventas, gráficas estadísticas, tabla de ventas recientes, visualización de tendencias de 
consumo.  
Interacción con 
otros módulos  Módulo de Ventas (fuente principal de datos), Módulo de Productos (información de productos 
y presentaciones), Módulo de Producción (apoyo para decisiones de producción), Módulo de 
Seguridad (control de acceso).  
Reglas de Negocio  1.- El dashboard solo mostrará información del día actual por defecto.  
2.- Los datos deben reflejar únicamente ventas confirmadas.  
3.- El producto más vendido se calcula en función de la cantidad total vendida.  
4.- La información se actualizará cada vez que se acceda al módulo.  
5.- El acceso a la información dependerá del rol del usuario.  
6.- En caso de no existir ventas, se mostrará información en cero o mensajes indicativos.

Página 9 de 63 RF03 — Gestión de Usuarios (Empleados)  
Campo  Descripción  
ID RF03  
Nombre  Gestión de Usuarios (Empleados)  
Propósito  Gestionar el ciclo de vida de los usuarios internos del sistema (empleados), permitiendo su 
registro, consulta, actualización, desactivación y búsqueda, garantizando un control adecuado de 
acceso y trazabilidad en el sistema SoftBakery.  
Público Objetivo  Área de Administradores (único rol autorizado para gestionar usuarios).  
Descripción 
General  El módulo permite al administrador gestionar los usuarios internos del sistema mediante 
operaciones CRUD (Crear, Leer, Actualizar y Desactivar). Los usuarios representan a los 
empleados que interactúan con los módulos de ventas, producción e inventario, y su acceso está 
controlado mediante roles.  
Características del 
Requerimiento  1.- Vista general: Visualización de lista de usuarios con información relevante.  
2.- Registro de usuarios: Alta de nuevos empleados con validación de campos obligatorios ( 
id_usuario, nombre_usuario, correo, password_hash, nombre, apellidos, teléfono, ciudad, 
estado, id_rol, activo y fechas de registro).  
3.- Contraseña segura: Validación de política de contraseñas (mínimo 8 caracteres, mayúscula, 
minúscula, número y carácter especial) y almacenamiento mediante hash seguro.  
4.- Edición de usuarios: Modificación de datos del usuario excepto el identificador (id_usuario). 
Desactivación lógica: Cambio de estado de activo a inactivo sin eliminación física del registro. 
Búsqueda de usuarios: Filtro por nombre o nombre de usuario.  
5.- Gestión de roles: Asignación de roles (Administrador, Ventas, Producción).  
6.- Validación de datos: Control de campos obligatorios y formatos correctos (correo, teléfono). 
Mensajes al usuario: Confirmaciones y errores claros.  
Flujo del Proceso  1.- El administrador accede al módulo de usuarios.  
2.- Visualiza la lista de empleados registrados.  
3.- Puede registrar un nuevo usuario ingresando los datos requeridos.  
4.- El sistema valida la información y guarda el usuario.  
5.- Puede editar información de usuarios existentes.  
6.- Puede desactivar usuarios (sin eliminarlos).  
7.- Puede buscar usuarios por nombre o usuario.  
Entradas  Nombre de usuario, contraseña, correo,  nombre, apellidos, teléfono, ciudad, estado, rol, estado 
(activo/inactivo).  
Salidas  Lista de usuarios, confirmaciones de registro/edición/desactivación, mensajes de error 
controlados.  
Interacción con 
otros módulos  Módulo de Login (autenticación), Módulo de Seguridad (control de acceso, logs), todos los 
módulos operativos (Ventas, Producción, Inventario) para validación de permisos según rol.  
Reglas de Negocio  1.- Solo el rol Administrador puede gestionar usuarios.  
2.- El nombre de usuario debe ser único en el sistema.  
3.- No se permite eliminar usuarios, solo desactivarlos.  
4.- Un usuario inactivo no puede acceder al sistema.  
5.- Todos los usuarios deben tener un rol asignado.  
6.- La contraseña debe cumplir con la política de seguridad establecida.  
7.- Se debe registrar la fecha de creación del usuario.

Página 13 de 63 RF04 — Gestión de Roles  
Campo  Descripción  
ID RF04  
Nombre  Gestión de Roles  
Propósito  Definir, gestionar y controlar los permisos y accesos de los diferentes tipos de usuarios del 
sistema SoftBakery, aplicando el principio de mínimo privilegio para garantizar la seguridad y 
correcta operación del sistema.  
Público Objetivo  Área de Administradores  
Descripción 
General  El módulo permite al administrador gestionar los roles del sistema mediante operaciones 
CRUD, definiendo los permisos específicos sobre los distintos módulos (usuarios, ventas, 
inventario, producción, etc.). Cada rol determina las acciones que un usuario p uede realizar 
dentro del sistema.  
Características del 
Requerimiento  1.- CRUD de roles: Creación, consulta, actualización y desactivación lógica de roles.  
2.- Campos del rol: id_rol, nombre_rol, descripción, estado_rol, fecha_creación, 
fecha_actualización. Definición de permisos: Asignación de permisos por módulo y acción 
(leer, crear, editar, desactivar).  
3.- Roles base: Administrador (acceso total), Ventas (registro de ventas y consulta), 
Producción (gestión de producción e inventario de materia prima).  
4.- Restricción de roles base: Los roles Administrador, Ventas y Producción no pueden ser 
desactivados.  
5.- Creación de roles: Permite registrar nuevos roles con nombre y descripción únicos.  
6.- Edición de roles: Permite modificar descripción y permisos, pero no el id_rol.  
7.- Desactivación lógica: Cambio de estado a inactivo sin eliminación física.  
8.- Validación de datos: Verificación de campos obligatorios y unicidad del nombre del rol.  
9.- Mensajes al usuario: Confirmaciones y errores claros en español.  
Flujo del Proceso  1.- El administrador accede al módulo de roles.  
2.- Visualiza la lista de roles existentes.  
3.- Puede crear un nuevo rol ingresando nombre y descripción.  
4.- Define los permisos por módulo y acción.  
5.- Puede editar roles existentes (excepto el id).  
6.- Puede desactivar roles (excepto roles base).  
7.- Los cambios se reflejan automáticamente en los accesos de los usuarios.  
Entradas  Nombre del rol, descripción, permisos por módulo (leer, crear, editar, desactivar), estado del rol.  
Salidas  Lista de roles, confirmaciones de registro/edición/desactivación, mensajes de error controlados.  
Interacción con 
otros módulos  Módulo de Usuarios (asignación de roles), Módulo de Login (validación de permisos), Módulo 
de Seguridad (control de acceso RBAC), todos los módulos del sistema (restricción de acciones 
según rol).  
Reglas de Negocio  1.- Solo el administrador puede gestionar roles.  
2.- El nombre del rol debe ser único.  
3.- No se pueden eliminar roles, solo desactivarlos.  
4.- Los roles base (Administrador, Ventas, Producción) no pueden ser desactivados.  
5.- Todo usuario debe tener un rol asignado.  
6.- Los permisos se asignan por módulo y tipo de acción.  
7.- Los cambios en roles afectan inmediatamente los accesos de los usuarios asociados.

Campo  Descripción  
ID RF12  
Nombre  Costo y Utilidad de Producto  
Propósito  Calcular el costo de producción de cada producto terminado y estimar su utilidad, permitiendo 
al área de administración tomar decisiones informadas sobre precios y rentabilidad en el 
sistema SoftBakery.  
Público Objetivo  Área de Administración  
Descripción General  El módulo permite calcular el costo total de producción de un producto a partir de su receta, 
considerando las cantidades de materia prima, su costo unitario y el porcentaje de merma. A 
partir de este costo, se calcula la utilidad y el porcentaje de gananc ia del producto.  
Características del 
Requerimiento  1.- Cálculo de costo por ingrediente:  Se calcula el costo de cada materia prima considerando 
cantidad requerida, costo unitario y merma aplicada.  
2.- Fórmula de costo real:  cantidad_real = cantidad_requerida × (1 + porcentaje_merma / 
100). Costo por ingrediente:  costo_ingrediente = cantidad_real × costo_unitario.  
3.- Costo total de producción:  Suma de los costos de todos los ingredientes de la receta.  
4.- Cálculo de utilidad:  utilidad = precio_venta − costo_produccion.  
5.- Porcentaje de utilidad:  porcentaje_utilidad = (utilidad / precio_venta) × 100.  
6.- Consulta de costos:  Visualización del costo y utilidad por producto.  
7.- Actualización dinámica:  Recalcula automáticamente cuando cambian precios de materia 
prima o receta.  
8.- Validación de datos: Verifica que existan costos definidos para materias primas.  
9.- Mensajes al usuario:  Resultados claros y comprensibles.  
Flujo del Proceso  1.- El sistema obtiene la receta del producto.  
2.- Recupera las materias primas y sus cantidades.  
3.- Aplica la merma para calcular la cantidad real.  
4.- Calcula el costo de cada ingrediente.  
5.- Suma los costos para obtener el costo total de producción.  
6.- Obtiene el precio de venta del producto.  
7.- Calcula la utilidad y el porcentaje de ganancia.  
8.- Muestra los resultados al usuario.  
Entradas  Receta del producto, cantidades de materia prima, porcentaje de merma, costo unitario de 
insumos, precio de venta.  
Salidas  Costo de producción, utilidad por producto, porcentaje de utilidad, desglose de costos por 
ingrediente.  
Interacción con 
otros módulos  Módulo de Recetas (cantidades), Módulo de Inventario (costos de materia prima), Módulo de 
Productos (precio de venta), Dashboard (indicadores financieros).  
Reglas de Negocio  1.- Todo producto debe tener una receta asociada.  
2.- Se debe aplicar el porcentaje de merma en el cálculo de costos.  
3.- El costo total es la suma de todos los ingredientes.  
4.- La utilidad depende del precio de venta definido.  
5.- El porcentaje de utilidad se calcula respecto al precio de venta.  
6.- Si no existe costo de materia prima, no se puede calcular el costo del producto.

Página 60 de 63 RF18 — Generar Pedido  
Campo  Descripción  
ID RF18  
Nombre  Generación de Pedido  
Propósito  Permitir al cliente confirmar su carrito, generar un pedido formal y registrar el pago (en línea o 
contra entrega), garantizando la trazabilidad del proceso desde la solicitud hasta su gestión 
interna.  
Público Objetivo  Clientes  
Descripción 
General  El módulo convierte el carrito de compras en un pedido formal, almacenando su detalle, 
permitiendo su seguimiento y registrando la información de pago. El pedido puede ser pagado en 
línea o contra entrega, manteniendo control del estado del pago y del pedi do. 
Características 
del 
Requerimiento  Campos del pedido:  id_pedido (automático), id_cliente, fecha_pedido, total_pedido (calculado), 
fecha_entrega, estado_pedido (pendiente/confirmado/pagado/entregado/cancelado).  
Campos de pago:  estado_pago (pendiente/pagado), tipo_pago (en línea/contra entrega) .  
Generación automática:  El pedido se crea a partir del carrito .  
Detalle del pedido:  Se almacenan los productos, cantidades y subtotales.  
Validación de stock:  Verifica disponibilidad antes de confirmar.  
Gestión de pago:  Si el pago es en línea, el estado_pago puede cambiar a pagado; si es contra 
entrega, permanece pendiente hasta confirmación manual.  
Consulta de pedidos:  El cliente puede visualizar sus pedidos y estados.  
Cancelación:  Solo disponible si el estado_pedido es pendiente.  
Mensajes al usuario:  Confirmaciones y errores claros.  
Flujo del 
Proceso  1. El cliente revisa su carrito (RF17).  
2. Selecciona tipo de pago (en línea o contra entrega).  
3. Confirma el pedido.  
4. El sistema valida stock y genera el pedido en estado PENDIENTE.  
5. Se guarda el detalle del pedido.  
6. Si el pago es en línea, se registra la referencia y se actualiza estado_pago a PAGADO.  
7. Si es contra entrega, el estado_pago queda en PENDIENTE.  
8. El pedido pasa a gestión interna (RF14).  
Entradas  Carrito, cliente, fecha de entrega, tipo de pago, referencia de pago (opcional).  
Salidas  Pedido generado, estado del pedido, estado del pago, historial de pedidos.  
Interacción con 
otros módulos  RF17 Carrito de Compras, RF14 Gestión de Pedidos, RF13 Ventas, RF11 Inventario de 
Productos.  
Reglas de 
Negocio  Todo pedido debe generarse a partir de un carrito con al menos un producto. El pedido inicia en 
estado PENDIENTE. El estado_pago inicia como PENDIENTE. Solo pedidos pendientes pueden 
cancelarse. Si el pago es en línea, debe registrarse una referencia de pa go. Si el pago es contra 
entrega, el pago se confirma al momento de la entrega. El stock debe validarse antes de generar 
el pedido. Todos los pedidos deben quedar registrados para trazabilidad.
